# 简历表述与面试问答

## 简历表述

智能售后工单处理系统｜FastAPI + AI 辅助分类业务项目

- 基于 FastAPI 实现售后工单处理后端，支持工单创建、列表查询、详情查看、状态更新和 SQLite 持久化。
- 设计工单业务字段和状态流转，包含分类、优先级、处理部门、AI 摘要和客服回复建议。
- 实现规则驱动的 AI mock fallback，根据工单标题和描述判断售后分类、优先级和处理部门；无 API key 时可完整本地演示。
- 预留 OpenAI 兼容接口调用入口，用于后续接入真实模型生成分类分析和回复建议。
- 编写 README、接口测试示例和 MySQL 建表 SQL，通过项目练习 Python 后端 CRUD、业务流程建模和 AI 辅助客服场景。

## 1 分钟介绍

我做了一个智能售后工单处理系统，基于 FastAPI 实现。它模拟企业客服售后场景，支持创建工单、查看列表和详情、更新状态、AI 辅助分类以及生成客服回复建议。默认使用 SQLite，本地可以直接运行，同时提供 MySQL 建表 SQL。AI 部分没有强依赖 API key，没有 key 时会根据关键词规则判断分类、优先级和处理部门，并生成模板回复；有 key 时预留 OpenAI 兼容调用入口。这个项目不是完整 Agent，也没有多轮对话，重点是展示 Python 后端、CRUD、状态流转和 AI 辅助业务处理能力。

## 高频问题

1. 这个项目解决什么问题？

它模拟售后客服处理工单的业务流程，帮助客服快速识别问题类型、优先级和处理部门，并生成回复建议。

2. 为什么做工单系统？

工单系统比普通 Demo 更贴近企业业务，能体现 CRUD、状态流转、业务字段建模和 AI 辅助处理。

3. 工单有哪些状态？

`new`、`processing`、`waiting_customer`、`resolved`、`closed`。

4. 状态流转怎么控制？

`workflow.py` 定义允许的状态转换，非法转换会返回 400。

5. AI 分类怎么做？

没有 API key 时使用规则匹配关键词，判断 category、priority、department；有 key 时预留 LLM 调用入口。

6. 为什么要 mock fallback？

为了保证项目没有 API key 也能本地运行和演示，避免核心业务被外部服务卡住。

7. 这个项目是 Agent 吗？

不是。它没有任务规划、多轮记忆、工具链自动选择，只是 AI 辅助分类和回复建议。

8. 为什么用 SQLite？

SQLite 适合本地 Demo，不需要安装数据库服务。项目提供 MySQL SQL，后续可以迁移。

9. 你真的用了 MySQL 吗？

当前默认实现是 SQLite。MySQL 只是提供建表 SQL，真实连接后才能写 MySQL 项目经验。

10. 如果继续优化，你会做什么？

先接真实 LLM 分类，再接 MySQL，然后补客服备注、处理记录、基础分页和简单前端。
